What Is a Distributed System?
Definition
A distributed system is a collection of autonomous computing elements
that appears to its users as a single coherent system.
Autonomous nodes:
Each node has its own processor, memory, and operating system
No shared physical memory between nodes
Nodes communicate exclusively through message passing
Cooperation among nodes creates the illusion of a single system

Distributed systems must manage:
Group membership (who is part of the system)

Authentication and trust management

The Core Idea: Tools vs. Rules
Distributed systems must separate ”how to do something” from
”deciding when to do it.”
Mechanism (The Tool):
The physical/technical capability.
Answers:”What is possible?”
Policy (The Rule):
The logic governing the capability.
Answers:”How and when should we use the tool?”
Naveen Kumar (SVNIT, Surat) Distributed Computing January 13, 2026 38 / 91
Step 1: The Mechanism (The Capability)
Mechanisms should be general and ”dumb”—they just follow orders.
Real Example: Smart Door Lock
Mechanism:The hardware that can retract or extend the bolt.
It doesn’t knowwhois at the door; it only knows how to lock/unlock.
Technical Example: Web Cache Storage
Mechanism:The functions to write data to RAM or delete it.
It doesn’t knowwhichfile is important; it only knows how to
save/delete.
Naveen Kumar (SVNIT, Surat) Distributed Computing January 13, 2026 39 / 91
Step 2: The Policy (The Decision)
Policies use the same mechanism to create different behaviors.
Smart Lock Policies:
Day Mode:”Stay unlocked if the owner is home.”
Night Mode:”Lock automatically at 10:00 PM.”
Web Cache Policies:
LRU Policy:”If full, delete the Least Recently Used file.”
FIFO Policy:”If full, delete the oldest file first.”
Naveen Kumar (SVNIT, Surat) Distributed Computing January 13, 2026 40 / 91
The Result: Modular Design
By separating them, we getPluggable Systems.
The ”Lego” Approach:
You can swap a ”Night Policy” for a ”Vacation Policy” without buying
a new lock.
You can upgrade a Cache Policy to be ”AI-driven” without rewriting
the code that actually saves files to the disk.
Changing a policy should be as simple as changing a configuration file.
Naveen Kumar (SVNIT, Surat) Distributed Computing January 13, 2026 41 / 91
The Trade-off: Complexity vs. Power
The Problem: Total separation creates ”Decision Overload.”
The Conflict:
If a system has 10 mechanisms and each has 5 policy options, the user
has 50 settings to manage.
This is ”Configuration Fatigue” (e.g., a professional server with
1,000-page manuals).
The Solution: Sensible Defaults
Ship the system with a ”Standard Policy” that works for 90% of people.
Keep the interface open so the 10% (experts) can plug in a custom
policy.

Replication, Caching, and Consistency
Replicationmaintains multiple copies of data or services to improve
availability, fault tolerance, and load balancing.
Cachingstores temporary local copies of data at clients or proxies to
reduce access latency and decrease network and server load.
Consistency Challenge:Replication and caching create multiple
data copies, and maintaining strong consistency requires immediate
update propagation and global agreement on operation order, which
significantly limits scalability.

ACID Properties Transactions
Atomicity: A transaction is executed entirely or not at all, with no
partial effects.
Consistency: A transaction transforms the system from one valid
state to another, preserving all rules and constraints.
Isolation: Concurrent transactions execute independently, without
interfering with each other.
Durability: Once a transaction commits, its effects persist despite
crashes or failures.

Enterprise Application Integration: Approaches
File transfer: Applications exchange data by producing and consuming shared files; simple
to implement but requires agreement on file formats, storage, naming, and update
notifications.
Shared database: All applications access a common database using a shared schema,
enabling easy data sharing and triggers, but risking schema complexity and performance
bottlenecks.
Remote Procedure Call (RPC): Applications invoke operations of other applications
directly, encapsulating behavior and triggering actions, but requiring both parties to be
active simultaneously.
Messaging: Applications communicate by exchanging messages via a middleware system,
allowing asynchronous interaction and decoupling, at the cost of message handling and
reliability concerns.
Application integration is inherently complex, and middleware plays a crucial role by
providing standardized RPC and messaging facilities.
