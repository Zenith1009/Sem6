Processes vs Threads: Conceptual Foundation




     A process is a program in execution with:
            Its own address space
            Strong OS-enforced isolation
            High context-switch overhead
     A thread is a lightweight execution unit within a process:
            Shares address space with other threads
            Has its own execution flow
            Faster switching, weaker protection

     OS represents execution using contexts
     Process context includes:
            Address space and memory mappings
            Open files and resources
            Privileges and accounting data
     Thread context includes:
            Program counter, registers, stack
            Minimal scheduling information

User-Level vs Kernel-Level Threads
     Threads can be managed at different levels of the system
     The management level determines performance and parallelism

User-Level Threads
     Managed by a user-space thread library
     Kernel is unaware of individual threads
     Thread creation and switching are fast
     A blocking system call blocks the entire process

Kernel-Level Threads
     Managed directly by the operating system kernel
     Kernel schedules each thread independently
     Supports true parallelism on multicore processors
     Thread operations incur higher overhead
 Naveen Kumar (SVNIT, Surat)   Distributed Computing     February 10, 2026   16 / 118
Thread Mapping Models
     Thread models define how user-level threads
     are mapped to kernel-level threads
     Many-to-One
            Multiple user threads mapped to a single kernel thread
            Implemented using user-level thread libraries
     One-to-One
            Each user thread mapped to a separate kernel thread
            Implemented using kernel-level threads
     Many-to-Many
            Multiple user threads multiplexed over multiple kernel threads
            Combines performance and parallelism

Interrupting a Server: Out-of-Band Communication
     Controlled interruption:
            A mechanism that allows a client to stop or control an ongoing server
            operation
            The connection is not abruptly terminated
     Out-of-band communication:
            Control messages that are handled with higher priority than normal
            data
     Why this is needed:
            Clients may want to cancel long-running operations
            Example: stopping a large file transfer
     Example: FTP
            FTP uses a separate control connection and a data connection
            Commands such as ABOR (abort) are sent on the control connection
            The server interrupts the data transfer without closing the session
     Alternative mechanism:
            TCP urgent (out-of-band) data sent on the same connection

Scalability: The ”Static” Problem
