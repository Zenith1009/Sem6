Communication in Distributed Systems


     Communication is the foundation of every distributed system
     Processes execute on different machines and must exchange
     information
     Early systems relied on low-level message passing
     Message passing is:
            Harder to program than shared memory
            Error-prone in large, unreliable networks (e.g., the Internet)
     Modern systems involve thousands to millions of processes
     Hence, scalable systems require higher-level communication
     abstractions

What Is a Remote Procedure Call?




     A process on machine A invokes a procedure on machine B
     The client process blocks until the result is returned
     Parameters are sent to the server
     The procedure executes remotely
     Results are returned to the client

Goal: Remote calls should be indistinguishable from local calls.

Parameter Passing in Local Calls



     Parameters are passed as representations, not abstractions
     Call-by-value (e.g., data)
            Actual value copied onto the stack
            Callee works on a local copy
            Modifications do not affect caller

     Call-by-reference (e.g., dbList)
            Memory address (pointer) is passed
            Callee accesses the same object as caller
            Modifications directly affect caller state




 Naveen Kumar (SVNIT, Surat)       Distributed Computing   February 15, 2026   29 / 99
Why Local Call Semantics Do Not Carry Over to RPC



     Pointers are only meaningful within one address space
     A memory address on machine A is meaningless on machine B
     Call-by-reference cannot be preserved directly
            Must be replaced by copying data
            Changes semantics compared to local calls

     Leads to fundamental RPC issues:
            Parameter marshaling
            Copy vs reference semantics
            Consistency and failure handling

Steps in a Remote Procedure Call
 1    The client procedure calls the client stub in the normal way
 2    The client stub builds a message and calls the local operating
      system
 3    The client’s OS sends the message to the remote OS
 4    The remote OS delivers the message to the server stub
 5    The server stub unpacks the parameters and calls the server
      procedure
 6    The server executes the procedure and returns the result to the stub
 7    The server stub packs the result into a reply message and calls its
      local OS
 8    The server’s OS sends the reply message to the client’s OS
 9    The client’s OS delivers the reply to the client stub
 10   The client stub unpacks the result and returns it to the client
      procedure

Marshaling and Unmarshaling




     Marshaling: The process of packaging procedure parameters into a
     message format suitable for transmission.
            Example: The client stub takes integers a and b and converts them
            into a byte stream: [proc: "doit", val(a), val(b)].
     Unmarshaling: The process of unpacking the message at the receiver
     to store the data back into its original data types.
            Example: The server stub receives the byte stream and extracts the
            values to call the local Implementation of doit.




 Naveen Kumar (SVNIT, Surat)      Distributed Computing       February 15, 2026   36 / 99
Data Representation: Endianness


     Problem: Different architectures store multibyte data (like integers)
     differently in memory.
     Big-Endian: Most significant byte (MSB) stored at the lowest
     address (e.g., IBM, Sun Sparc).
     Little-Endian: Least significant byte (LSB) stored at the lowest
     address (e.g., Intel x86).
     Example: The value 0x12345678
            Big-Endian: [12][34][56][78]
            Little-Endian: [78][56][34][12]
     Solution: Convert to Network Byte Order (standardized as
     Big-Endian) during marshaling.



 Naveen Kumar (SVNIT, Surat)    Distributed Computing    February 15, 2026   37 / 99
The Problem with Pointers




     Isolated Address Spaces: A pointer is just a memory address (e.g.,
     0x004F) valid only within the client’s process.
     Meaningless Transfer: Passing 0x004F to a server is useless
     because the server’s memory at that address contains unrelated data
     or is inaccessible.
     Distributed Memory: There is no shared memory in a standard
     RPC; the ”stack” and ”heap” are physically separate.




 Naveen Kumar (SVNIT, Surat)   Distributed Computing   February 15, 2026   38 / 99
Solutions to Passing References


     Copy/Restore (Call-by-Value-Result):
            Copy the entire referenced object to the server.
            Server modifies it and sends the whole object back to overwrite the
            client’s local copy.
     Serialization: Flatten complex structures (linked lists, trees) into a
     contiguous byte stream.
     Remote Object References: Use a unique global ID that the server
     can use to ”call back” the client (standard in Java RMI).
     Restriction: Simply forbid passing pointers in the Interface Definition
     Language (IDL).

DCE RPC: Interface and Stub Generation


DCE RPC starts with defining a remote interface. A UUID is generated
to uniquely identify this interface.
The programmer writes an IDL file. It lists remote procedures and their
parameters. Only syntax is defined, not behavior.
The IDL compiler processes the IDL file. It generates:
     A shared header file
     A client stub
     A server stub

These stubs hide all communication details.




 Naveen Kumar (SVNIT, Surat)   Distributed Computing     February 15, 2026   57 / 99
DCE RPC: Building and Running an RPC



Client code is compiled with the client stub. Server code is compiled with
the server stub. Both include the same header file.
Object files are linked with the RPC runtime library. This produces the
client and server binaries.
At run time, the client calls a local procedure. The client stub sends a
request to the server. The server stub calls the real server code. The result
is returned transparently.
RPC makes remote calls look local.

     In real distributed systems, these assumptions may not hold
            Components may be temporarily unavailable
            Blocking reduces scalability

     Message-oriented systems adopt asynchronous communication
            Sender proceeds without waiting
            Looser coupling between sender and receiver

Limitations of Basic Sockets



     Low-level & Error-prone: Developers must manually manage IP addresses, ports,
     and connection states.
     Byte-Stream vs. Message: No built-in message boundaries; the application must
     manually track where data ends.
     Manual Marshaling: Requires explicit ”pickling” and ”unpickling” for every data
     exchange.
     Rigid Topology: Limited support for advanced communication patterns like
     Multicasting or Load Balancing.
     Tight Coupling: Blocking calls (accept, recv) reduce overall system scalability.
     Brittleness: If a server is unavailable during connect(), the system often fails
     without built-in recovery.

ZeroMQ: Request–Reply Pattern

The Request–Reply pattern is the most basic messaging model in ZeroMQ, designed to
mimic traditional client–server interactions.

     Socket Roles:
            REQ (Request): Used by the Client to initiate a conversation by sending a
            message.
            REP (Reply): Used by the Server to receive requests and send back specific
            responses.

ZeroMQ: Publish–Subscribe Pattern

The Publish–Subscribe pattern enables one-to-many communication, decoupling the
sender from the receivers.

     Socket Roles:
            PUB (Publisher): Broadcasts messages to all connected endpoints
            simultaneously.
            SUB (Subscriber): Connects to publishers to listen for specific data streams.
     Topic-Based Filtering: Subscribers use prefixes (tags) to filter incoming traffic;
     messages not matching the tag are discarded by the middleware before reaching
     the application.
     Loose Coupling:
            Space: Publishers don’t know who or how many subscribers exist.
            Time: New subscribers can join or leave the network at any time.
     Transient Nature: Messages are not queued for offline subscribers; if no one is
     listening when a message is published, it is lost forever.

MPI: Parallel Computing Model
MPI provides a high-level, hardware-independent standard for tightly-coupled
parallel applications.

     Why MPI?: Sockets are too low-level; MPI is optimized for specialized
     interconnects and performance over general TCP/IP.
     Addressing: Uses (groupID, processID) instead of IP addresses; failures
     are considered fatal.
     Flexible Semantics: Supports varying performance needs:
            Blocking: MPI send / MPI recv (wait for buffer/data).
            Non-blocking: MPI isend / MPI irecv (immediate return).
            Synchronous: MPI ssend (wait until transmission starts).
